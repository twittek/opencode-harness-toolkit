Adaptive Scenario Discovery

The interview works like an Akinator-style diagnostic system: Bayesian inference updates scenario probabilities, while information-gain-based questioning asks the next question that reduces uncertainty the most.

Not a static questionnaire.
A Bayesian bootstrapping interview.

Bayesian inference

Field: probability theory and Bayesian statistics. Every answer updates the likelihood of possible development scenarios.

Information theory

Concepts: entropy reduction and information gain. The next question is chosen because it reduces uncertainty the most.

Decision trees

Field: machine learning, computer science and statistics. Irrelevant branches are pruned as soon as they become unlikely.

❌

Weak early question

Do you use Oracle Spatial?

This is too narrow. It only helps after the interview already knows that GIS or spatial data is relevant.

✅

High-value early question

Are there external systems or integrations?

This partitions the scenario space early and decides whether integration policies, scripts or MCP discovery stay active.

∑

Mathematical intuition

A good question maximizes expected information gain and minimizes entropy in the remaining scenario distribution.

Bayesian inference updates beliefs.
Information theory chooses the next best question.
Decision-tree pruning removes irrelevant branches.
Active learning makes the interview adaptive.
One question.
One answer.
Update the scenario model.
Prune irrelevant branches.
Ask the next best question.

Integration & Tooling Discovery

The adaptive interview discovers external systems early, because integrations strongly shape what the agent may read, write, automate or avoid.

🔎

High-information question

Are there external systems or tools the agents should know about?

This single question decides whether issue tracking, code review, docs, design, quality, monitoring, wrapper scripts or MCP planning stay active.

🧭

System categories

GitLab, GitHub, Jira, Confluence, Azure DevOps, Linear, Figma, SonarQube, Sentry, Grafana, Prometheus and more.

🛡️

Safety model

Unknown systems have no access. Write actions require approval. Secrets are never stored in harness files.

✅

Wrapper-first

Agents should use approved scripts instead of inventing ad-hoc CLI calls for write operations.

⚙️

MCP-aware

The toolkit can document MCP candidates and policies, but does not install MCP servers during harness initialization.

📌

Policy-driven

Each system gets purpose, access method, read/write boundaries, approval rules and forbidden operations.

Expanded Role Model

The harness can generate specialist role guidance and activation rules so the agent applies the right expert perspective at the right time.

🧠

Expert lenses

🎯

Activation policy

Roles are activated only when relevant. The harness avoids role noise and uses the smallest useful role set.

⚖️

Conflict handling

When perspectives conflict, the policy prioritizes security, correctness and maintainability over speed and convenience.

Context Loading Policy

Large context windows are powerful, but the harness should still load only the smallest useful context set.

🧠

128K optimized

The policy supports large-context models without encouraging the agent to load the whole harness for every task.

📚

Structured knowledge space

The harness is modular knowledge: baseline files, relevant policies, specialist roles and explicit run history.

🧹

Context hygiene

Generated folders, dependencies, backups, logs and historical run artifacts are not loaded automatically.

Tier 1: always load baseline context
Tier 2: load relevant roles, integrations and policies
Tier 3: load run artifacts only on explicit need
Tier 4: never load node_modules, dist, build, coverage, template, backups or logs

Self-Verification Policy

Agents must verify their own work against the original requirement before reporting completion.

✅

Requirement match

Every task ends with a check that the generated result actually satisfies the user's request.

🧪

Checks and tests

Agents should run available validation steps or explicitly say why they were not run.

⚠️

Honest completion

Remaining assumptions, risks or skipped checks must be reported instead of hidden.

MCP Discovery & Planning

/harness-mcp discovers MCP opportunities, evaluates risk and creates approval-first installation plans.

🧩

Capability gaps

The command identifies where MCP could help: repositories, work tracking, documentation, design systems, browser automation or local tooling.

⚖️

Risk classification

Servers are classified as read-only, write-capable, browser-control, filesystem, secret-sensitive, production-adjacent or unknown.

✅

Approval first

No MCP server is installed or configured automatically. The command writes recommendations and asks before changing config.

/harness-mcp

1. Discover capability gaps
2. Recommend MCP candidates
3. Review risk
4. Create installation plan
5. Ask for approval
6. Apply config only after approval

Config safety

Everything you need to operate the harness

A small lifecycle command set keeps responsibilities clean: initialize, inspect, update, and improve from experience.

01
/harness-init

Generate the project harness

Selectable guided setup, one question at a time. German interview, English harness files, roles, playbooks, scripts, safety defaults, versioning, and changelog.

02
/harness-check

Audit the current version

Finds drift, contradictions, missing files, duplicate rules, unsafe shell patterns, and context bloat. Writes active findings.

03
/harness-update

Apply approved findings

Processes active findings for the current version, requests approval, applies changes, updates changelog, and increments version.

04
/harness-retro

Improve from real usage

Captures satisfaction, pain points, start/stop/change feedback, and turns real experience into structured findings.

Governance that does not get in the way

The toolkit is intentionally lightweight: Markdown files, shell scripts, versioned findings, and explicit changelogs. No heavy platform required.

🔐

Safety defaults

  • No merge or deployment by default
  • No dependency changes without approval
  • No file deletion without approval
  • No test weakening
📋

Versioned findings

Check and retro findings are scoped to the current harness version, so stale recommendations do not silently mutate newer governance.

🧾

Changelog discipline

Every harness change is recorded with source, applied findings, changed files, safety implications, and follow-up items.

Command decision matrix
